Jag känner fortfarande att det är knepigt med UML, man jobbar helt enkelt alldeles för lite med det. När det kommer till hur man strukturerar UML så finns det flera olika typer av relationer, tre utav dem kanske vanligaste är Aggregation, Komposition och Association.

Om man t.ex. har två klasser som studenter och lärare som båda använder sig utav varanda dvs. en student kan ha flera olika lärare och en lärare kan ha flera olika studenter då visualiserar men detta med en rak linje emellan klasserna för att visa på att det är Association.

Skulle man däremot införa ytterliggare en klass Avdelningar med en relation till klassen lärare då pratar vi om en Aggregation. Här börjar vi prata om "ägande" dvs. att avdelningar äger lärare. En avdelning kan ha flera lärare och en lärare kan tillhöra flera avdelningar. Detta visas som en rak linje med en ofylld diamand som pekar mot den ägande klassen. Man skulle kunna säga att det är löst ägande eftersom båda klasserna kan existera utan varandra.

Om man däremot inför ytterligare en klass Universitet där Universitet är i stark relation med Avdelningar. Avdelningar kan med andra ord bara tillhöra ett Universitet och ett Universitet kan ha flera Avdelningar. Detta visas med en rak linje med en ifylld diamant som pekar mot den ägande klassen. Detta är ett starkt ägande då Avdelningar inte kan existera utan ett Universitet. Detta kallas för en Kompositions relation.

Började med Queue uppgiften och måste säga det att det ändå tog ett litet tag att förstå det hela med hur man använder Node objekt till att göra en liknelse utav Python's inbyggda lista. Ganska snabbt förstod man att, med hjälp utav att rita upp Node objekten och på så sätt få visualisera listan, gick allt mycket enklare. Strukturen på den skapade listan följer en linjär data struktur nämligen en Cirkulär Enkellänkad lista. Detta eftersom den sista noden är länkad till den första "head" noden.

Nästa uppgift var att skapa en lista, en Unordered list. Tycker det mesta påminner om ovan uppgift där jag skapade en queue. Bara att man använder sig utav andra methoder. Efter att ha gjort queu uppgiften gick denna uppgift väldigt lätt att göra, då man nu fått en bra förståelse för hur man kopplar Node objekten med varandra. Trodde det skulle vara svårare än vad det var att lägga till nodes i mitten utav listan samt ta bort nodes på valda positioner pga. ombindning utav nodernas länkarna.

Under queue uppgiften skapade jag ett exception nämligen:

  • EmptyQueueException

    • Triggas då man försöker kolla current value samt next value på en tom lista.

    Under list uppgiften finner man 2 exceptions nämligen:

  • IndexErrorException

    • Kastas om man försöker sätta in ett värde på ett index som inte exsisterar. Om man försöker byta ut ett värde på ett index som inte finns. Om man försöker hämta ett värde från ett index som inte finns.

  • ValueErrorException

    • Kastas om man försöker hämta ett index från ett värde som inte finns eller om man försöker ta bort ett värde som inte finns.

    Som svar på vad en länkad lista är kontra en array handlar nog svaret mest om hur dem hanterar sina index. En länkad lista har egentligen inga index utan är uppbyggd utav länkar som länkar varje nod med nästa nod. I en array däremot är värdena inte länkade med varandra utan dem tillhör bara sina respektive index.

    Egentligen första gången jag sätter mig in i rekursiva funktioner. Är nästan helt säker på att jag använt mig utav det tidigare men läste aldrig på och satte mig in i termen. Efter lite research om rekursiva funktioner har jag bara sett varningar att man inte ska använda sig utav det och mestadels verkar det handla om att det är slött och mer memory samt processor krävande.

    Men åter till frågan, en rekursiv funktion är en funktion som på nått sätt anropar sig själv direkt eller via en annan funktion. Ett exempel på detta kan se ut som följande:

  • - for-loop
    • for i in range(0,5) print(i)
  • - recusive
    • def rec_range(n): if not n <= 1: return rec_range(n-1) + (n-1,) return (0,)

    Kanske inte det bästa exemplet men det funkar och påvisar skillnaden.

    Det var inte allt för svårt att förstå hur sorteringsalgoritmer fungerar. Finns algoritmer går igenom listan och jämför intilliggande par med varandra och om dem ligger i fel ordning byter dem plats med varandra, proceduren pågår tills listan är helt sorterad (Bubble sort). Bubble sort kanske inte är att föredra det finns två andra varianter man i så fall bör använda nämligen Selection sort och Insertion sort där den sistnämnda är snabbast. Insertion sort fungerar i stora drag precis som när du sorterar dina kort på handen under ett kortspel. Man tar ett kort ittererar genom korten på handen och placerar det där den hör hemma och tillslut har man en sorterad hand.

    VisuAlgo är riktigt bra inte bara för att visualisera sorteringsalgoritmer utan också många andra typer utav datastrukturer. När det kommer till sorteringsalgoritmerna på sidan, är det ett perfekt verktyg att få se hur allt fungerar både visuelt och hur det hanteras i kod. Denna sida hamnade direkt i bibloteket inför framtiden.

    För att enkelt förklara big-O, så är det ett "mätsätt" som på ett ungefärligt sätt beräknar hur lång tid en funktion tar på sig för att beräkna sin utdata.

    Kursmomentet motsvarde ungefär föregående kursmoment, gick lätt att komma igång och var helt lagom storleksmässigt.
